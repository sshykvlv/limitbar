# LimitBar — дизайн-спека

**Дата:** 2026-07-11 · **Статус:** на ревью

## Что это

Опенсорсная (MIT) минималистичная macOS-менюбарка, показывающая остаток лимитов
по нескольким AI-аккаунтам сразу: N аккаунтов Claude (у владельца два Max 20x)
плюс Codex (ChatGPT-подписка). Замена ClaudeUsageBar, у которого один аккаунт
и ручная вставка куки.

**Референс стиля и архитектуры — Lidless** (`~/dev/mac-keep-awake`): AppKit,
`NSStatusItem` + стандартный `NSMenu`-дропдаун, минимум файлов, `build.sh` со
swiftc, собственный лёгкий чекер апдейтов, никаких сторонних зависимостей.

## Цели / не-цели

**Цели**
- Все аккаунты видны одним взглядом в стандартном выпадающем меню.
- Ноль ручной возни с куками: основной Claude и Codex подхватываются сами,
  дополнительные Claude-аккаунты — через браузерный OAuth-логин один раз.
- Супер-нативный вид: системные шрифты, SF Symbols, template-иконка,
  автоматическая светлая/тёмная тема, стандартное поведение меню.

**Не-цели (v1)**
- История использования, графики, стоимость токенов.
- Провайдеры кроме Claude и Codex (архитектура допускает, UI не обещает).
- App Store, sandbox. Дистрибуция — GitHub Releases + brew tap.
- Нотификации о порогах (кандидат на v1.1).

## UX

### Иконка в меню-баре — мини-индикатор

Одна иконка, но она сама — индикатор: кастомно отрисованный «мини-эквалайзер»,
по одному тонкому вертикальному столбику на аккаунт (порядок как в меню:
Claude-аккаунты, потом Codex). Высота заливки столбика = **сколько осталось**
в самом тесном окне аккаунта (`1 − max(5h, weekly)/100`). Трек столбика —
слабая подложка, чтобы пустой уровень тоже читался.

- Монохромный template-рендер — выглядит как системные иконки звука/Wi-Fi,
  сам адаптируется к светлой/тёмной теме.
- Если у какого-то аккаунта осталось < 10% — его столбик рисуется
  `systemRed` (иконка на этот случай рендерится non-template с цветами,
  снятыми с текущего appearance, и перерисовывается при смене темы).
- Геометрия: столбик ~3.5 pt, зазор 2.5 pt, высота 15 pt на канвасе 18 pt;
  3 аккаунта → иконка ~16 pt шириной.

### Меню (стандартный NSMenu)

```
┌──────────────────────────────────────────────┐
│ Max основной    5h ▓▓▓▓▓▓░░░░ 62%   wk ▓▓▓░ 31% │  ← кастомная вьюха-строка
│ Max второй      5h ▓▓▓▓▓▓▓▓▓░ 91%   wk ▓▓▓▓ 48% │
│ Codex           5h ▓░░░░░░░░░ 12%   wk ▓▓░░ 23% │
├──────────────────────────────────────────────┤
│ Updated 12:04 · Refresh            ⌘R        │
│ Add Claude Account…                          │
├──────────────────────────────────────────────┤
│ Launch at Login                              │
│ Check for Updates…                           │
│ View on GitHub                               │
├──────────────────────────────────────────────┤
│ Quit LimitBar                      ⌘Q        │
└──────────────────────────────────────────────┘
```

- **Строка аккаунта** — `NSMenuItem` с кастомной вьюхой (`NSMenuItem.view`,
  внутри `NSHostingView` со SwiftUI-строкой): имя аккаунта, два капсульных
  бара (5-часовое окно и неделя) с процентами. Моноширинные цифры
  (`monospacedDigit`), системный шрифт, высота строки как у системных меню
  с кастомными вьюхами (Wi-Fi, Battery).
- **Цвет баров**: системный секондари → `systemYellow` (>70%) →
  `systemRed` (>90%).
- **Время до сброса**: второй строкой мелким `secondaryLabelColor` текстом
  под баром, только когда утилизация окна ≥ 50% (иначе шум). Полное время
  сброса обоих окон — в tooltip строки.
- **Клик по строке аккаунта** — подменю: `Rename…` (NSAlert с текстовым
  полем), `Re-login…` (для протухшего аккаунта), `Remove`.
- **Refresh при открытии**: `menuWillOpen` дёргает обновление (как в
  Lidless), плюс фоновый поллинг.
- Порядок строк: Claude-аккаунты в порядке добавления, Codex последним.

## Провайдеры и авторизация

### Claude

- **Эндпоинт**: `GET https://api.anthropic.com/api/oauth/usage` с
  `Authorization: Bearer <access_token>`. Возвращает утилизацию (%) и время
  сброса для окон `five_hour` и `seven_day`.
- ⚠️ **Обязателен** заголовок `User-Agent: claude-code/<актуальная версия>` —
  без него запрос попадает в агрессивно лимитируемый бакет (перманентные 429).
- **Основной аккаунт — автоподхват**: читаем OAuth-креды Claude Code из
  Keychain (запись `Claude Code-credentials`, JSON с access/refresh-токенами).
  Read-only: свой рефреш для этой записи не делаем, чтобы не гонки с Claude
  Code; если access протух — перечитываем Keychain (Claude Code сам его
  обновляет). Если и после перечитывания протух (Claude Code давно не
  запускался) — бейдж «⚠︎ open Claude Code» на строке.
- **Дополнительные аккаунты — встроенный OAuth PKCE** (тот же публичный
  client_id, что у Claude Code):
  1. «Add Claude Account…» → поднимаем локальный HTTP-листенер на
     loopback-порту → открываем в браузере страницу авторизации claude.ai
     с PKCE-челленджем и redirect на localhost;
  2. пользователь логинится под нужным аккаунтом, жмёт Authorize;
  3. браузер редиректит на localhost, ловим code, меняем на
     access+refresh-токены;
  4. фолбэк: если авто-редирект не сработал — поле «вставь код со страницы».
- **Рефреш** дополнительных аккаунтов — свой, по refresh_token, в фоне,
  за 5 минут до истечения access. Невалидный refresh → строка живёт с
  бейджем «⚠︎ re-login», клик открывает OAuth-флоу заново.
- **Идентификация аккаунта**: e-mail из ответа token-эндпоинта/профиля,
  отображаемое имя пользователь задаёт сам (Rename).

### Codex

- **Автоподхват**: токены из `~/.codex/auth.json` (уже есть у владельца).
  Следим за файлом через DispatchSource — перелогин в codex CLI подхватывается
  сам. Рефреш-токен из auth.json используем так же, как codex CLI; обновлённый
  access назад в файл не пишем (файл — собственность CLI, мы read-only).
- **Лимиты**: 5h/weekly окна с бэкенда ChatGPT — берём подход, задокументированный
  в CodexBar (`docs/codex.md` их репо): запрос к usage-эндпоинту бэкенда с
  access-токеном из auth.json. Точный путь эндпоинта фиксируем на этапе
  имплементации по актуальному коду CodexBar (он живой и MIT).
- Нет `~/.codex/auth.json` → строка Codex не показывается вовсе (не ошибка).

## Архитектура

Стиль Lidless — минимум файлов, но не один (логики больше):

```
Sources/
  main.swift            — AppDelegate, NSStatusItem, сборка NSMenu
  MenuRows.swift        — SwiftUI-строка аккаунта + NSHostingView-обёртка
  Poller.swift          — таймер (60 c), fan-out по аккаунтам, backoff
  Providers/
    UsageProvider.swift — протокол: fetchUsage(account) async throws -> Usage
    ClaudeProvider.swift
    CodexProvider.swift
  Accounts.swift        — AccountStore: список аккаунтов, персист
  KeychainStore.swift   — секреты (свои токены; чтение записи Claude Code)
  OAuthFlow.swift       — PKCE-флоу + локальный listener
  Updates.swift         — чекер апдейтов (портируем из Lidless)
build.sh / release.sh / Info.plist — по образцу Lidless
```

**Модель данных**

```swift
struct Usage { let fiveHour: Window; let sevenDay: Window }
struct Window { let utilization: Double; let resetsAt: Date }
struct Account { let id: UUID; var name: String; let kind: Kind
                 enum Kind { case claudeMain, claudeOAuth, codex } }
```

- Метаданные аккаунтов (id, имя, kind, порядок) — UserDefaults.
- Секреты (access/refresh дополнительных Claude-аккаунтов) — Keychain,
  по записи на аккаунт (`limitbar.<uuid>`).
- Приложение не сандбоксится (нужен `~/.codex/auth.json` и чужая
  Keychain-запись); подписываем Developer ID + нотарификация.

## Поллинг и ошибки

- Интервал 60 с (не настраивается, YAGNI) + мгновенный refresh на
  `menuWillOpen`, но не чаще раза в 10 с на аккаунт.
- 429 → экспоненциальный backoff (2, 4, 8 мин, потолок 15) для этого
  аккаунта, остальных не трогает.
- Сеть недоступна → показываем последние данные, таймстемп «Updated …»
  сереет и показывает возраст («Updated 25 min ago»).
- Любая ошибка аккаунта видна только бейджем на его строке — меню никогда
  не пустое и не модальное, никаких алертов кроме явных действий пользователя.

## Дистрибуция и апдейты

- GitHub public repo `limitbar`, MIT.
- `build.sh` (swiftc, universal binary), `release.sh` — zip + GitHub Release,
  cask в существующем `~/dev/homebrew-tap`.
- Самообновление — портируем механизм Lidless (тихая проверка при старте +
  пункт «Check for Updates…»), без Sparkle.

## Тестирование

- Парсинг ответов провайдеров и логика порогов/бэкоффа — unit-тесты на
  фикстурах JSON (swift test, лёгкий Package.swift рядом не заводим —
  тестовый таргет собирается тем же build.sh с XCTest либо простым
  assert-раннером; решается в плане имплементации).
- OAuth-флоу и Keychain — ручной чек-лист (реальные аккаунты владельца).
- Верификация перед релизом: оба Max-аккаунта + Codex показывают цифры,
  совпадающие с `/usage` в Claude Code и codex CLI.

## Разрешённые хвосты (сняты 2026-07-11 перед планом)

1. **Codex usage-эндпоинт**: `GET https://chatgpt.com/backend-api/wham/usage`,
   `Authorization: Bearer <access_token из auth.json>`;
   `rate_limit.primary_window` → 5h, `secondary_window` → неделя
   (источник: CodexBar docs/codex.md).
2. **Keychain-запись Claude Code** (проверено на живой записи): JSON с ключом
   `claudeAiOauth` → `{accessToken, refreshToken, expiresAt (ms epoch),
   scopes[], subscriptionType, rateLimitTier}`. Возможен вариант записи только
   с `mcpOAuth` без `claudeAiOauth` (Claude Code 2.1.x гоча) — трактуем как
   «нет основного аккаунта», не как крэш.
3. **Claude usage-эндпоинт**: обязателен также заголовок
   `anthropic-beta: oauth-2025-04-20`; токену нужен scope `user:profile`
   (источник: CodexBar docs/claude.md).
4. Название: рабочее **LimitBar** (репо `limitbar`) — владелец может
   переименовать до публикации.
